Visitor és Observer minta

Design Patterns C++ példán keresztül

Visitor Pattern Observer Pattern

Bemutatkozás. Két tervezési mintát fogunk együtt látni egy komplex példán keresztül. Mindkettőt már ismerhetik külön-külön, most az együttes alkalmazást látjuk.

Mit fogunk látni?

  • Visitor minta: Új művelet hozzáadása meglévő osztályokhoz azok módosítása nélkül
  • Observer minta: Automatikus értesítés állapotváltozásról
  • Együttes alkalmazás: Elemeket tárolunk, módosítjuk őket visitorokkal, és az observerek értesülnek a változásokról
  • Gyakorlati példa: int és string értékek tárolása és manipulálása
Két pattern együtt - a Visitor új műveleteket ad, az Observer értesít. Konkrét példa: ElementContainer tárol elemeket, Visitor műveleteket hajt végre rajtuk, Observer figyeli őket.

VisitorBase ősosztály

UML

classDiagram class VisitorBase { <<abstract>> +Visit(ElementInt) +Visit(ElementString) } class ElementInt class ElementString VisitorBase ..> ElementInt VisitorBase ..> ElementString

Forráskód


class VisitorBase
{
public:
    virtual void Visit(ElementInt& element) = 0;
    virtual void Visit(ElementString& element) = 0;
};

A visitor leszármazottai végrehajtanak műveleteket az elemeken. Minden elemtípushoz külön Visit metódus.

Visitor alapelve: minden elemtípushoz külön Visit metódus. Az elemek meghívják a visitort önmagukra (double dispatch).

ObserverBase ősosztály

UML

classDiagram class ObserverBase { <<abstract>> +ValueChanged(int, int) +ValueChanged(string, string) }

Forráskód


class ObserverBase
{
public:
    virtual void ValueChanged(int oldValue, 
                            int newValue) = 0;
    virtual void ValueChanged(string oldValue, 
                            string newValue) = 0;
};

Az observerek értesülnek az elemek állapotváltozásáról. Overloading: különböző típusokhoz külön metódus.

Observer pattern: publish-subscribe mechanizmus. Amikor változik egy elem értéke, minden regisztrált observer értesül.

ElementBase ősosztály


class ElementBase
{
protected:
    vector<ObserverBase*> observers;
public:
    virtual void Show() = 0;
    virtual void Accept(VisitorBase& visitor) = 0;

    void RegisterObserver(ObserverBase *observer)
    {
        observers.push_back(observer);
    }
};
  • Tárolja a regisztrált observereket
  • Accept: visitor minta kulcsa (double dispatch)
  • Show: megjelenítés a konzolon
Ez az alap minden elem számára. Observers vektor tárolja a figyelőket. Accept metódus fog visszahívni a visitorra.

ElementInt osztály


class ElementInt : public ElementBase
{
    int value;
public:
    int GetValue() { return value; }
    void SetValue(int newValue)
    {
        int oldValue = value;
        value = newValue;
        for(auto observer : observers)
        {
            observer->ValueChanged(oldValue, newValue);
        }
    }

    ElementInt(const int value = 0) : value(value) {}

    virtual void Accept(VisitorBase& visitor) override
    {
        visitor.Visit(*this);
    }

    virtual void Show() override
    {
        cout << "ElementInt: " << value << endl;
    }
};
Figyeljük meg: SetValue értesíti az összes observert. Accept visszahívja a visitort, így automatikusan a megfelelő Visit hívódik.

ElementString osztály


class ElementString : public ElementBase
{
    string value;
public:
    string GetValue() { return value; }
    void SetValue(string newValue)
    {
        string oldValue = value;
        value = newValue;
        for(auto observer : observers)
        {
            observer->ValueChanged(oldValue, newValue);
        }
    }

    ElementString(const string value) : value(value) {}

    virtual void Accept(VisitorBase& visitor) override
    {
        visitor.Visit(*this);
    }

    virtual void Show() override
    {
        cout << "ElementString: " << value << endl;
    }
};

⚠️ Azonos szerkezet, csak más típussal - ez a visitor minta ereje!

Szinte ugyanaz, mint az ElementInt, csak string típussal. A visitor automatikusan a megfelelő típust kapja.

Konkrét Visitor implementáció


class Visitor : public VisitorBase
{
public:
    virtual void Visit(ElementInt& element) override
    {
        element.SetValue(element.GetValue() + 100);
    }

    virtual void Visit(ElementString& element) override
    {
        element.SetValue(element.GetValue() + "+100");
    }
};

Visitor minta előnyei

  • Új művelet = új visitor osztály
  • Nem kell módosítani az Element osztályokat
  • Típusbiztos (fordítási időben ellenőrzött)
Ez a művelet: +100 minden int értékhez, "+100" string minden stringhez. Ha új műveletet akarunk, csak új visitor osztályt írunk, az elemeket nem bántjuk.

Konkrét Observer implementáció


class Observer : public ObserverBase
{
public:
    virtual void ValueChanged(int oldValue, 
                            int newValue) override
    {
        cout << "Int value changed from " 
             << oldValue << " to " 
             << newValue << endl;
    }

    virtual void ValueChanged(string oldValue, 
                            string newValue) override
    {
        cout << "String value changed from " 
             << oldValue << " to " 
             << newValue << endl;
    }
};

Egyszerű implementáció: kiírja a változásokat a konzolra. Több observer is lehet, mindegyik értesül!

Ez egy egyszerű observer, csak konzolra ír. Valós alkalmazásban lehet logger, UI frissítő, bármi.

ElementContainer osztály


class ElementContainer
{
    vector<std::unique_ptr<ElementBase>> elements;
public:
    void AddAndTakeOwnership(
        std::unique_ptr<ElementBase>& element)
    {
        elements.push_back(std::move(element));
    }

    void Show()
    {
        for(auto& element : elements)
        {
            element->Show();
        }
    }

    void Accept(VisitorBase& visitor)
    {
        for(auto& element : elements)
        {
            element->Accept(visitor);
        }
    }
};
unique_ptr: automatikus memóriakezelés, ownership szemantika. std::move: az ownership átvitele, másolás nem lehetséges. Accept továbbítja a visitort minden elemnek.

Smart Pointer részletek

unique_ptr és move semantics

  • unique_ptr = exkluzív ownership
  • Nem másolható, csak mozgatható (std::move)
  • Mozgatás után az eredeti pointer nullptr
  • Automatikus memóriafelszabadítás

// Így működik:
unique_ptr<ElementBase> elem = make_unique<ElementInt>(42);
container.AddAndTakeOwnership(elem);  // elem átadva
// elem már nullptr, container birtokolja az objektumot
elem = make_unique<ElementString>("new"); // újrahasznosítható!
Modern C++ fontos eleme. Biztonságos memóriakezelés, nincs memory leak. Move semantics miatt hatékony, nincs felesleges másolás.

Főprogram - elemek létrehozása


ElementContainer container;
Observer observer;

unique_ptr<ElementBase> newElement = make_unique<ElementInt>(1);
newElement->RegisterObserver(&observer);
container.AddAndTakeOwnership(newElement);

newElement = make_unique<ElementString>("1");
newElement->RegisterObserver(&observer);
container.AddAndTakeOwnership(newElement);

newElement = make_unique<ElementInt>(2);
newElement->RegisterObserver(&observer);
container.AddAndTakeOwnership(newElement);

newElement = make_unique<ElementString>("2");
newElement->RegisterObserver(&observer);
container.AddAndTakeOwnership(newElement);

Érdekes: A newElement újrahasznosítható, mert a move után nullptr-re áll!

4 elem: 2 int, 2 string. Mindhez regisztráljuk az observert. A newElement változó újrahasznosítása mutatja a move szemantika előnyét.

Főprogram - teljes futás


cout << "--- Before visit:" << endl;
container.Show();
--- Before visit:
ElementInt: 1
ElementString: 1
ElementInt: 2
ElementString: 2 

cout << "--- During visit:" << endl;
Visitor visitor;
container.Accept(visitor);
--- During visit:
Int value changed from 1 to 101
String value changed from 1 to 1+100
Int value changed from 2 to 102
String value changed from 2 to 2+100 

cout << "--- After visit:" << endl;
container.Show();
--- After visit:
ElementInt: 101
ElementString: 1+100
ElementInt: 102
ElementString: 2+100
Látjuk a teljes folyamatot: kezdeti értékek → visitor módosít → observerek értesülnek → végeredmény. A visitor SetValue-t hív, ami trigger-eli az observereket.

Összefoglalás

Visitor Pattern

  • Új művelet meglévő osztályokhoz
  • Osztályok módosítása nélkül
  • Double dispatch
  • Típusbiztos

Observer Pattern

  • Automatikus értesítés
  • Publish-subscribe
  • Lazán csatolt komponensek
  • Több observer is lehet

Együttes alkalmazás előnyei

A Visitor flexibilis műveletvégzést biztosít,
az Observer pedig átlátható állapotkezelést.
Modern C++ technikákkal (smart pointers) biztonságos megoldás.

Két pattern együtt használva erőteljes eszköz. Visitor: bővíthetőség. Observer: reaktivitás. Smart pointerek: biztonság.

Ellenőrző kérdések

Teszteljük a megértést! (Kattints a válaszhoz)

1. Kérdés

Mi a Visitor minta fő előnye?

Válasz: Új művelet hozzáadása meglévő osztályokhoz azok módosítása nélkül. Új visitor osztályt írunk, az Element osztályokat nem kell megváltoztatni. Ez követi az Open-Closed Principle-t.

2. Kérdés

Hogyan és mikor értesülnek az observerek az értékváltozásról?

Válasz: A SetValue metódus végigmegy az observers vektoron és meghívja mindegyik observer ValueChanged metódusát az új és régi értékkel. Ez azonnal történik, amikor az érték változik.

3. Kérdés

Miért fontos az Accept metódusban a visitor visszahívása?

Válasz: Ez a "double dispatch" mechanizmus lényege. Az Accept meghívja visitor.Visit(*this), ami miatt automatikusan a megfelelő típusú Visit metódus hívódik (ElementInt → Visit(ElementInt&), ElementString → Visit(ElementString&)). Így fordítási időben típusbiztos.

4. Kérdés

Mit csinál a unique_ptr move semantika?

Válasz: A std::move átadja az ownership-et a célnak. Az eredeti unique_ptr nullptr-re áll, a cél pedig átveszi az objektum birtoklását. Így garantált, hogy mindig pontosan egy unique_ptr birtokolja az objektumot, és nem lehet véletlenül sem lemásolni.

5. Kérdés

Milyen előnye van a két minta együttes alkalmazásának ebben a példában?

Válasz: A Visitor lehetővé teszi új műveletek (pl. +100, *2, log, export stb.) hozzáadását az elemekhez kód duplikáció és Element módosítás nélkül. Az Observer pedig automatikusan értesít minden változásról (logging, UI frissítés, validáció stb.), így a komponensek lazán csatoltak maradnak. Együtt: rugalmas, bővíthető és reaktív rendszer.
